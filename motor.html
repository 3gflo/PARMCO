Raspberry Pi Control Backend

The real-time C/ASM embedded software that drives the DC motor over hardware PWM, measures tachometer speed, and dynamically regulates power based on Bluetooth RFCOMM commands.

System Architecture (C Backend)

The primary executable, bt_motor_control.c, initializes the BCM2835 hardware library and acts as a central control loop. It relies on a two-layer loop architecture to guarantee low-latency hardware control while serving network requests:

  • Outer Loop (Connection Handling): Blocks on accept() waiting for the Android app to connect. If the connection drops, it safely resets the motor state (forcing PWM to 0) and loops back to wait for a reconnection.
  • Inner Loop (High-Speed Control): Polls the IR sensors for RPM calculations with a 100µs delay to prevent 100% CPU lockup. Crucially, it uses the non-blocking select() system call on the Bluetooth socket; incoming commands are parsed instantly without ever stalling the motor control logic.

ARM Assembly Feedback Controller

To strictly control closed-loop logic execution, the adaptive proportional (P) controller is written in 32-bit ARM Assembly (feedback.S) using the AArch32 calling convention. Instead of floating-point math, it uses high-speed bit-shifting for dynamic gain scheduling.

Adaptive Gain Scheduling

The controller adjusts the "P-Gain" dynamically based on the absolute error size. Massive errors trigger an aggressive division by 8 (Shift 3), while smaller errors gracefully step down to divisions of 16, 32, and finally 64.

Deadband & Dampening

At an error < 100, the Shift 6 division acts as a deadband, preventing the motor from violently oscillating. If the target speed is low (< 1000 RPM), an extra shift layer is added to prevent over-torquing the system.

Safety Clamping

After the offset is calculated and applied to the current PWM, the assembly routine rigorously clamps the final output so it never drops below 0 or exceeds the hardware limit of 1000.

Headless Deployment & Build Tools

To ensure the Raspberry Pi can operate as an isolated embedded system, interaction with the desktop environment is entirely removed.

  • Headless Bluetooth: The system utilizes a custom systemd service (bluetooth-agent.service) running /usr/bin/bt-agent -c NoInputNoOutput. This forces the BlueZ stack to automatically accept all incoming pairing requests without prompting for a PIN.
  • Compilation: The provided makefile links both the C application and the compiled assembly object with -lbcm2835 (for hardware GPIO) and -lbluetooth (for the RFCOMM socket).
# Compilation via Makefile gcc -Wall -O2 -o bt_motor_control bt_motor_control.c feedback.S -lbcm2835 -lbluetooth

Design Limitations

As documented in the design analysis, this implementation has a few constraints:

  • Polling-Based RPM: Because the BCM2835 userspace library lacks true interrupt-driven edge counting, the system relies on tight polling. Extremely high RPM pulses shorter than the 100µs polling window could theoretically be missed.
  • Feedback Interval: The control loop only updates the PWM offset once per second. Transient load disturbances occurring within that one-second window are not corrected until the next cycle.
  • Steady-State Error: The lack of Integral (I) and Derivative (D) terms means a small steady-state error can persist when the bit-shifted deadband divisor becomes larger than the remaining RPM gap.
