Android App Implementation

A Kotlin-based application providing real-time wireless control and telemetry monitoring of the Raspberry Pi 4 motor control system over Bluetooth RFCOMM.

Architecture & Threading Model

The app follows a single-Activity architecture where MainActivity.kt manages UI binding, state management, and Bluetooth I/O.

  • Background Threading: The Bluetooth socket connection and telemetry listener run on a dedicated background thread spawned inside the connectToDevice() function.
  • UI Updates: Because Android UI cannot be updated from background threads, telemetry data parsing uses runOnUiThread{} blocks to safely dispatch MEASURED_RPM and EXTERNAL_RPM updates back to the main thread.
  • Connection Lifecycle: The listener thread blocks on inputStream.read(). If the connection drops, it catches an IOException, updates the UI to "Disconnected", and safely kills the listening thread.

User Interface (activity_main.xml)

The layout is contained within a ScrollView to accommodate all device screen sizes and uses a vertical LinearLayout for structure. Key sections include:

Connection & Status

A centered dynamic text view (tvConnectionStatus) displays the current Bluetooth state, alongside a full-width "Connect to Pi" button.

Motor Controls

A horizontal layout pairs the Start/Stop toggle button with a "Reverse" polarity switch. A RadioGroup beneath it handles the mutually exclusive control modes: Manual, Maintain, and Synced.

Live Telemetry

The bottom of the screen features dedicated text views for intended PWM, Measured RPM (styled in bold red), and External RPM (styled in bold blue).

Bluetooth Handshake & Synchronization

The app handles legacy and modern Bluetooth permissions seamlessly, requesting BLUETOOTH_CONNECT and BLUETOOTH_SCAN on Android 12+ (API 31).

When connecting, the app creates a secure RFCOMM socket using the standard Serial Port Profile (SPP) UUID: 00001101-0000-1000-8000-00805F9B34FB. Immediately upon connection, the app executes a State Sync—it blasts its current UI state (mode, intended PWM, polarity, target RPM) to the Pi so both devices begin in perfect agreement.

Communication Protocol

All communication uses newline-delimited plaintext strings.

App → Pi (Commands)

Command Description
STATE:START / STATE:STOPEnables or disables motor output (forces PWM to 0).
DIR:FORWARD / DIR:REVERSESets H-bridge polarity.
MODE:<MANUAL/MAINTAIN/SYNCED>Switches the active control logic.
PWM:<0-1000>Sets raw duty cycle in Manual mode.
TARGET_RPM:<value>Sets the desired RPM for the feedback controller.

Pi → App (Telemetry)

Message Description
MEASURED_RPM:<value>Controlled motor speed from IR tachometer.
EXTERNAL_RPM:<value>External master motor speed from second IR sensor.

Known Limitations & Constraints

  • No automatic reconnection: If the Bluetooth connection is severed, the user must manually trigger a reconnection; the app does not poll to reconnect on its own.
  • Unbounded RPM Input: The EditText field for Target RPM does not restrict values on the Android side. However, the Pi's feedback loop clamps the resulting PWM output safely.
  • Single-thread parsing: The socket reader uses a 1024-byte buffer. While it gracefully splits chained strings via newline delimiters, partial messages split across separate physical reads are not explicitly reassembled.
